# Manual: LOD generation (xLODGen/SSELODGen + TexGen + DynDOLOD)

This is a maintainer-grade, reproducible manual for generating LOD outputs locally in Mod Organizer 2 (MO2).

Scope: Skyrim Special Edition, MO2-based workflow, Tamriel first pass.

Policy reminder: these outputs are **user-generated** and **local-only**. Do not commit or ship them in this repo. See `docs/generated-outputs-policy.md`.

## Purpose (what each tool does, and why the order matters)

Skyrim LOD is not one thing:

- **Terrain (and water) LOD**: the distant landscape shapes/textures (mountains, ground, coastlines).
  - Generated by **xLODGen / SSELODGen** (xEdit LODGen mode).
- **Object / tree LOD**: distant buildings, rocks, ruins, trees, etc, plus the generated LOD plugin(s).
  - **TexGenx64.exe** generates LOD textures (including billboards) used by DynDOLOD.
  - **DynDOLODx64.exe** generates object/tree LOD and the LOD plugin(s).

Important: **DynDOLOD does not generate terrain (or water) LOD**. Terrain is handled in the xLODGen/SSELODGen step.

## Prerequisites and non-negotiable rules

- **Freeze your modlist + plugin list + load order** before generation.
  - Any meaningful change to textures/meshes/trees/landscape can require re-generation.
- **Generated outputs stay local**:
  - Do not commit.
  - Do not redistribute.
  - Do not move outputs into your real Skyrim game folder.
- **Run generators through MO2** and keep Overwrite clean:
  - Let the tool write into MO2 Overwrite.
  - Immediately “Move content to mod…” into a dedicated output mod.
- **Use dedicated MO2 output mods** (consistent naming):
  - `VR - Output - xLODGen`
  - `VR - Output - TexGen`
  - `VR - Output - DynDOLOD`

## MO2 setup (recommended structure)

Left pane separators (suggested):

- `---- VR - Resources ----`
- `---- VR - Outputs (Local, Generated) ----`

Create these empty mods up front (left pane, under Outputs):

- `VR - Output - xLODGen`
- `VR - Output - TexGen`
- `VR - Output - DynDOLOD`

Recommended left pane ordering (high level):

- Resources and required “input” mods first.
- Output mods late, near the end, with a stable order:
  - `VR - Output - xLODGen`
  - `VR - Output - TexGen`
  - `VR - Output - DynDOLOD`

Rule: outputs should win conflicts only when that is the intended result (they are built from the rest of the stack).

## Tool setup (MO2 executables)

Add these tools as executables in MO2 and always run them through MO2:

- `TexGenx64.exe`
- `DynDOLODx64.exe` (note: **not** “DynDOLODDx64.exe”)
- `xLODGen` / `SSELODGen` (name varies)
  - This is xEdit running in “LODGen” mode. Treat it as “xLODGen” in documentation even if the executable is named SSELODGen.

Placement recommendation:

- Install/run tools from a simple tools folder (example: `D:\Tools\...`), not from `Program Files`, not from the Steam game directory.

## Generation order (basic first pass used for Milestone 9)

### 1) Terrain LOD (Tamriel) with xLODGen / SSELODGen

Goal: generate **terrain** LOD only. Do **not** generate object LOD or tree LOD in this pipeline.

Why: DynDOLOD will generate object/tree LOD later, and splitting responsibilities reduces duplication and troubleshooting surface.

Steps (basic):

1. In MO2, enable your normal mod stack for play.
2. If you are using the temporary terrain-restore resource:
   - Enable the mod `xLODGen Resource - SSE Terrain Tamriel`.
   - Enable its plugin `SSE-Terrain-Tamriel.esm` (right pane) **only for this step** if your setup requires it.
3. Run `xLODGen` / `SSELODGen` from MO2.
4. In the worldspace list, select:
   - **Tamriel**
5. In options, check **only**:
   - **Terrain LOD**
6. Run generation and let it finish.
7. In MO2, right-click **Overwrite** → **Move content to mod…** → choose `VR - Output - xLODGen`.
8. Critical cleanup:
   - Disable `SSE-Terrain-Tamriel.esm` after terrain generation (see troubleshooting below).

### 2) TexGen (LOD textures for DynDOLOD)

Goal: generate the texture set DynDOLOD will use.

Steps (basic):

1. Ensure `VR - Output - xLODGen` is enabled (terrain LOD can coexist; it is not used by DynDOLOD directly, but keep outputs consistent).
2. Run `TexGenx64.exe` from MO2.
3. Use the basic/default preset.
4. Billboard choices for the basic pass:
   - Tree billboards: **ON**
   - HD / Rendered tree billboards: **OFF** (future enhancement)
   - Grass billboards: **OFF** (unless you are explicitly doing grass LOD)
5. Let it finish.
6. In MO2, move Overwrite into `VR - Output - TexGen`.

Sanity check: `VR - Output - TexGen` is enabled and placed so DynDOLOD “sees” it (i.e., it loads before `VR - Output - DynDOLOD`).

### 3) DynDOLOD (object/tree LOD + plugins)

Preflight (do this before launching):

- Ensure `VR - Output - TexGen` is enabled.
- Ensure `SSE-Terrain-Tamriel.esm` is **disabled** (it will block DynDOLOD).

Steps (basic):

1. Run `DynDOLODx64.exe` from MO2.
2. Use the basic/default preset (safe defaults).
3. Generate for the intended worldspace(s) (basic pass: Tamriel).
4. Let it finish and close DynDOLOD when done.
5. In MO2, move Overwrite into `VR - Output - DynDOLOD`.
6. Enable the generated plugin(s) as required by your setup and keep their order stable.

## 4K default guidance (conservative and reproducible)

Goal: ensure the generators target your actual display/output resolution, without “mystery” ultra settings.

- Confirm your game is configured for **3840x2160** (example: `SkyrimPrefs.ini` values `iSize W=3840` and `iSize H=2160`, or equivalent launcher/BethINI configuration).
- Expect higher VRAM usage at 4K. If you hit instability or stutter, reduce settings before assuming “it’s broken”.

DynDOLOD (Advanced) tile-size guidance for 2160p (use as a starting point, not a requirement):

- LOD texture size: **2048**
- Full texture size: **512**
- Billboard texture size: **2048**

Notes:

- Higher settings do not create detail that is not present in the source textures/meshes; they mostly change sampling and distance appearance.
- Keep the first pass consistent and repeatable; tune later only when you can measure the impact.

## Troubleshooting (known issues and fixes)

### DynDOLOD refuses to run because `SSE-Terrain-Tamriel.esm` is enabled

Symptom:

- DynDOLOD stops early and reports that `SSE-Terrain-Tamriel.esm` must not be active / is in the load order.

What it is:

- `SSE-Terrain-Tamriel.esm` is part of the **temporary** “xLODGen Resource - SSE Terrain Tamriel” terrain-restore workflow.

Why it happens:

- DynDOLOD explicitly blocks running with that plugin active.

Fix (MO2):

1. Open MO2 → right pane → **Plugins** tab.
2. Find `SSE-Terrain-Tamriel.esm`.
3. Uncheck/disable it.
4. Re-run `DynDOLODx64.exe`.

Rule: enable it only for the terrain LOD step if you truly need it, then disable it immediately afterward.

### Outputs landed in MO2 Overwrite

This is expected if you run through MO2.

Fix:

- Right-click **Overwrite** → **Move content to mod…**
- Move into the correct output mod:
  - `VR - Output - xLODGen`
  - `VR - Output - TexGen`
  - `VR - Output - DynDOLOD`

### Tool naming confusion

- `DynDOLODx64.exe` is correct. “DynDOLODDx64.exe” (double D) is a typo.
- `SSELODGen` vs `xLODGen`:
  - Treat them as “xLODGen” in the workflow; they are the same practical step (xEdit LODGen mode) in this pipeline.

## Validation checklist (quick in-game sanity)

- Load an exterior save with clear sightlines.
- Check a few wide vistas:
  - No purple/missing textures in the distance.
  - Terrain LOD looks continuous (no obvious seams or “holes” at range).
- Check forest edges and city exteriors:
  - Distant objects/trees appear plausibly (reduced pop-in).
  - No obvious missing billboard textures.
- If you changed anything significant afterward (trees/landscape/textures), treat this as invalidated and re-generate.

## Future enhancements (deliberate, optional upgrades)

- **Ultra Trees / Ultra mode**
  - Typically increases TexGen billboard work and DynDOLOD complexity and VRAM use.
  - Consider only after the basic pass is stable and you can measure the tradeoff.
- **Grass LOD**
  - Requires grass billboards and more time; adds another axis of troubleshooting.
  - Treat as a separate iteration (do not mix into a “first pass”).
- **Additional worldspaces**
  - Add beyond Tamriel only when you have a reason and a repeatable test route.
- **Performance/quality tuning**
  - Make one change at a time, regenerate only what changed, and keep notes on VRAM and frametime impact.
