# Iteration 0.1.x Plan (Fun UX Expansion, Controlled Risk)

## Goal

Deliver a UX-focused expansion cycle that is visibly more fun to play, while staying within VistulaRim stability constraints and preparing release `0.2.0`.

## Why this revision

The earlier `0.1.x` draft was documentation-heavy and too conservative for a game-facing expansion cycle. This revision shifts emphasis to player-visible UX improvements (HUD clarity, faster interactions, stronger combat feedback, better navigation, better first-run flow) while preserving release discipline.

## Scope guardrails (kept)

1. Keep Patch Pack assumptions unchanged (no redistribution of third-party assets).
2. Preserve stability-first validation and rollback paths.
3. Keep `0.1.x` at 10 milestones to stay aligned with the current versioning contract (`0.1.1` -> `0.1.10` -> `0.2.0`).
4. Push risky or speculative UX ideas to optional modules first.

| `0.1.6` | Combat feedback and quick-access fun pass | Combat readability package (`TrueHUD`) + quick action access candidate (`Wheeler`, optional) | Combat info is more legible and quick-access flow is stable in playtest |
| `0.1.7` | Widescreen/ultrawide hardening | Widescreen fix stack + tested presets + conflict list | Core menus/HUD usable at `2560x1080` and `3440x1440` |
| `0.1.8` | First-run onboarding + MCM automation | Quickstart path, missing-master triage, MCM preset workflow (`MCM Recorder`) | New setup reaches playable state in one guided pass |
| `0.1.9` | `skyrim-llm-runtime` prep (non-visible) | Integration prerequisites + interface constraints + event hook candidates | Contract exists for future visible integration, with zero runtime dependency in baseline |
| `0.1.10` | Release hardening + package prep | Full regression matrix (including fun UX scenarios), release checklist, technical release notes draft | Wabbajack package smoke-tested on Ubuntu 24.04 + Windows path; `0.2.0` ready |

## Execution policy for this iteration

1. Prefer high-adoption UX mods first, then experimental options.
2. Introduce at most one higher-risk UX element per milestone.
3. Keep optional/fun additions isolated behind clear module boundaries.
4. Reject additions that require heavy gameplay rebalance in this iteration.

## Completion gates

1. Every milestone has a matching execution log in `docs/execution/0.1/`.
2. At least one clearly player-visible UX improvement is delivered in milestones `0.1.2` through `0.1.8`.
3. `CHANGELOG.md` remains player-facing only.
4. `docs/releases/0.2.0.md` is complete before tagging.
5. Cross-platform smoke checks are attached to `0.1.10` execution notes.
